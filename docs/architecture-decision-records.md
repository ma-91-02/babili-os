# Architecture Decision Records (ADR) — Babili OS

> هذا الملف يوثق القرارات المعمارية الأساسية للمشروع حتى لا تضيع أسباب القرارات مستقبلًا.

---

# ADR-001 — اعتماد Web-first Architecture

## القرار
اعتماد Web-first كبداية رسمية للمشروع.

## السبب
- تسريع التطوير
- تقليل التعقيد
- سهولة التوسع عالميًا
- تقليل تكلفة الصيانة
- دعم أي جهاز ومتصفح

## النتيجة
- النظام يعمل من المتصفح مباشرة
- لا يوجد اعتماد إجباري على التطبيقات بالبداية
- يمكن إضافة تطبيقات الهاتف لاحقًا

---

# ADR-002 — اعتماد Multi-tenant Architecture

## القرار
اعتماد Multi-tenant Architecture منذ البداية.

## السبب
- دعم آلاف المطاعم عالميًا
- تقليل تكلفة البنية التحتية
- سهولة الإدارة
- سهولة التوسع

## النتيجة
- كل مطعم يملك بياناته الخاصة
- العزل المنطقي إلزامي
- يجب منع أي Data Leakage بين المطاعم

---

# ADR-003 — اعتماد Microservices Architecture

## القرار
تقسيم النظام إلى Microservices.

## السبب
- قابلية التوسع
- فصل المسؤوليات
- سهولة التطوير المستقبلي
- دعم فرق متعددة
- تقليل الترابط

## النتيجة
- API Gateway إلزامي
- كل Service مسؤولة عن Domain محدد
- Communication standards مطلوبة

---

# ADR-004 — اعتماد QR-first Dining Flow

## القرار
اعتماد QR كطريقة الدخول الأساسية داخل المطاعم.

## السبب
- تقليل الاحتكاك
- دعم السياح
- عدم الحاجة لتطبيق إجباري
- سهولة التبني عالميًا

## النتيجة
- Guest Ordering مدعوم
- تجربة سريعة بدون تسجيل
- QR جزء أساسي من تجربة المطعم

---

# ADR-005 — اعتماد Platform Brand وليس White-label

## القرار
بابلي منصة عالمية موحدة وليست نظام White-label.

## السبب
- بناء Network Effects
- بناء Brand عالمي
- توحيد التجربة
- بناء Social Graph عالمي

## النتيجة
- المطاعم تعمل داخل منظومة بابلي
- التجربة موحدة عالميًا
- Brand consistency إلزامية
